# Card Draft UI 설계 가이드

Card Draft UI의 목표는 플레이어가 매 화면에서 세 가지를 바로 이해하게 만드는 데 있다. 첫째, 지금 무엇을 해야 하는가. 둘째, 이번 런이 어떤 빌드로 성장 중인가. 셋째, 지금 선택이 덱을 어떻게 강하게 만드는가.

메인 메뉴는 단순한 시작 화면이 아니라 허브다. `이어하기`와 `새 런 시작`이 가장 먼저 보여야 하고, 왼쪽은 핵심 진입 버튼, 가운데는 영웅과 현재 런 분위기, 오른쪽은 현재 런 정보와 최근 런 기록, 빌드 통계를 보여주는 대시보드 구조가 맞다. 플레이어는 여기서 계속할지, 새로 시작할지를 바로 결정할 수 있어야 한다.

맵 화면은 전략 선택 화면이다. 이동 가능한 노드는 밝게, 지난 노드는 완료 상태로, 아직 못 가는 노드는 어둡게 보여야 한다. 왼쪽 범례는 초보자용 해설이고, 오른쪽 패널은 현재 목적지와 예상 보상을 설명한다. 하단 빌드 요약은 이번 런의 방향을 계속 상기시키는 역할을 한다.

카드 보상 화면은 가장 중요한 선택 화면이다. 카드 3장을 중앙에 크게 두고, 왼쪽에는 현재 빌드 요약, 오른쪽에는 보상과 추천 이유, `건너뛰기`를 둔다. 핵심은 좋은 카드 자체보다 내 덱에 맞는 카드가 무엇인지 바로 판단하게 해주는 것이다.

상점 화면은 구매 판단 화면이다. 상단에는 골드, 중앙에는 상품 목록, 한쪽에는 카드 제거와 덱 확인 같은 정비 기능을 둔다. 상품은 가격, 이름, 타입, 효과, 태그가 보여야 하며, 돈이 부족하면 즉시 알 수 있어야 한다.

전투 화면은 메인 플레이 화면이다. 가장 위에는 현재 목표와 다음 행동이 항상 보여야 한다. 중앙 전장은 적 5칸, 플레이어 5칸의 필드전 구조를 유지하고, 좌측은 적 영웅 정보, 우측은 로그와 주요 버튼, 하단은 손패와 플레이어 상태를 담당한다. 사용 가능한 카드와 공격 가능한 유닛은 강하게 강조해야 한다.

이벤트 화면은 런의 변수 구간이다. 상황 설명, 분위기 이미지, 선택지, 보상 미리보기가 한 세트로 읽혀야 한다. 플레이어는 선택 전부터 체력 손실, 골드 소모, 카드 강화, 유물 획득 같은 결과를 예측할 수 있어야 한다.

휴식 화면은 정비 구간이다. 전투보다 조용하고 안전한 분위기 안에서 회복, 강화, 진행을 고르게 해야 한다. 핵심은 지금 체력을 챙길지, 덱을 더 날카롭게 만들지를 판단하게 하는 것이다.

런 결과 화면은 한 판의 결산이다. 승패를 크게 보여주고, 플레이 시간, 처치 수, 획득 골드, 방문 노드, 최종 빌드, 주요 카드와 유물을 요약한다. 마지막에는 `메인 메뉴`와 `새로운 런`이 바로 보여야 한다.

공통 원칙은 네 가지다. 모든 화면은 다음 행동을 가장 먼저 보여준다. 빌드 정보는 맵, 보상, 전투, 결과에서 반복 노출한다. 선택의 이유를 짧게 설명한다. 화면마다 가장 중요한 정보의 우선순위를 다르게 둔다.

구현 기준은 `TopBar / MainContent / SidePanel / BottomAction` 구조다. 화면 모양은 달라도 이 틀을 유지하면 플레이어가 빠르게 익숙해진다. Card Draft UI는 화려함보다 유도력과 빌드 인지력이 우선이다.

## Godot 구현 메모

현재 코드 기준으로 UI를 개선할 때 가장 중요한 것은 `Container를 어디까지 쓰고, 어디부터 직접 배치할지`를 구분하는 것이다.

현재 구조:

- 전체 화면 셸: `ScrollContainer + VBoxContainer`
- 공통 패널/버튼/칩: `src/ui/ui_factory.gd`
- 전장 행: `HBoxContainer`
- 손패: `HFlowContainer`
- 카드 내부 텍스트 스타일: `style_card_title()`, `style_card_rules()`
- 카드 프레임 자산: `assets/ui/fantasy_card_frame.png`
- 공용 패널 자산: `assets/ui/fantasy_ui_panel.png`

현재 구현에서 잘 쓰고 있는 부분:

- 화면 단위 레이아웃은 `VBoxContainer`, `HBoxContainer`, `GridContainer`, `HFlowContainer`로 빠르게 정리한다.
- 카드/버튼 hover는 `mouse_entered`, `mouse_exited`, `create_tween()`으로 처리한다.
- 카드에 마우스를 올렸을 때 확대, 앞으로 튀어나오기, 툴팁 노출까지 연결되어 있다.
- 카드 텍스트는 그냥 Label 한 장으로 끝내지 않고 `이름`, `핵심 효과`, `원문 보조 설명`, `상태 바`로 나눈다.

현재 아직 부족한 부분:

- 손패가 아직 컨테이너 정렬형이라, 실제 카드게임처럼 부채꼴로 손에 들린 느낌은 약하다.
- 카드 드래그 중 목표 지점 프리뷰, 스냅 이동, 손패 재배치 애니메이션은 아직 없다.

권장 다음 단계:

1. 손패 부모를 `Control`로 바꾸고 카드 위치와 회전을 스크립트로 계산한다.
2. 중앙 카드만 크게 띄우고 양옆 카드는 살짝 회전시키는 부채꼴 배치를 적용한다.
3. hover 시 `scale`, `y offset`, `rotation reset`, `z_index`를 함께 바꾼다.
4. 드래그 중에는 가장 가까운 슬롯 또는 타깃을 하이라이트한다.

즉, Godot에서는 `모든 것을 컨테이너로만 해결하려고 할수록 카드 게임 UI가 개발툴처럼 보이기 쉽다.` 현재 프로젝트도 이 원칙을 기준으로 손패 영역을 후속 개선 대상으로 둔다.

## 카드 한 장의 권장 구조

현재 코드 기준 카드 한 장은 아래 구조가 가장 안정적이다.

- 바깥 프레임: `Button` 또는 `PanelContainer`
- 상단 헤더: 이름 + 비용 + 추천/사용 가능 배지
- 서브라인: 종족 / 속성
- 일러스트: `TextureRect`
- 핵심 효과: 짧은 요약 Label
- 보조 설명: 원문 설명 Label
- 하단 상태 바: `지금 사용 가능`, `마나 부족`, `공격 가능` 같은 즉시 상태

이 구조를 지키면 카드가 작아져도 `무슨 카드인지`, `지금 쓸 수 있는지`, `쓰면 뭐가 되는지`가 바로 읽힌다.
